k3s 上线闭环 ✅：滚动发布与回滚
我完成了一个端到端上线闭环：构建并发布镜像 v1/v2，在 k3s 中部署 Deployment + Service(NodePort)， 通过 Caddy 暴露 80 端口对外提供访问。项目重点是发布可控、切换可验证、问题可定位，体现运维与 DevOps 能力。
打开 Demo（/demo） 查看发布记录（文本） 代码仓库（GitHub） 服务运行中
Docker 镜像 v1/v2 k3s / Kubernetes Deployment（2 副本） Service(NodePort:30080) Rolling Update / Rollback Caddy :80 入口 UFW 收口端口
我负责的内容
从“能跑”做到“可上线、可发布、可回滚、可排障”。
交付与实现
  • 镜像版本管理：构建并发布 v1/v2 镜像，用于验证发布与回滚链路。
  • 集群编排：在 k3s 落地 Deployment + Service(NodePort)，保障副本数与滚动更新策略生效。
  • 公网访问：Caddy 监听 80，/ 为作品页，/demo 反向代理到 127.0.0.1:30080。
  • 发布控制：通过 kubectl set image 驱动更新，结合 rollout status/history 追踪发布结果。
运维与可用性
  • 可观测输出：页面展示 Deployment / Pods / 发布记录 / /demo 响应，便于核对真实运行状态。
  • 安全加固：UFW 只放行 22/80，封闭 NodePort 与 k8s 关键端口；禁用 root 远程登录，仅允许 dev 登录。
  • 排障路径：可从入口层（Caddy）→ NodePort → Pod 逐层用 curl/kubectl 定位问题。
发布可控 可回滚 可排障
发布与回滚能力（我能做到什么）
  • 滚动更新：更新镜像版本后，k3s 按策略逐步替换 Pod，服务连续可用。
  • 回滚：可切回上一版本镜像，快速恢复到稳定状态。
  • 发布记录：为每次发布增加 change-cause，保留历史修订信息。
相关输出在「运行状态」区实时展示（Deployment/Pods/Rollout）。
架构（访问链路）
从浏览器到 Pod：入口层与服务层解耦，便于定位与扩展。
定位思路：先看入口（Caddy / 80），再看 NodePort，再看 Pods 与镜像版本。
运行状态（实时）
页面直接读取服务器输出，用于核对当前部署与发布记录。
